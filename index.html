Task 1: Define Epics

Based on the goals, stakeholder needs, and requested features, identify 3 Epics that represent the major feature groups or outcomes for the MVP.

Creating Epics:

  • An Epic should be broad enough to include 3–7 user stories, but not so vague it loses focus.
  • If you find a cluster of related user stories, group them under a single Epic. Think of an Epic as a container that makes backlog management more manageable and thematic.
  • It's ok to start with User Stories first and then come back to catergorize you User Stories into Epics. Agile isn't prescriptive - it's what works best for YOU

Task 2: Write User Stories

For the Epics you created, write 4 User Stories. Ensure your stories are meeting the INVEST criteria!

Format: "As a [user], I want to [action] so that [benefit]"

Hints for Writing

  • Avoid generic phrases like "As a user." Be specific as to which user (e.g. patient, provider, admin, etc.)
  • Include the Why! Instead of saying "so that I can use the app," say "so that I can follow up on my treatment plan"
  • Keep it small. Each story should be able to be completed in 1 sprint.

INVEST criteria:

  • Independent: If stories rely on each other, consider combining them or re-sequencing
  • Negotiable: If the story is overly detailed with technical specs or UI mockups, it’s not negotiable
  • Valuable: Ask “Who benefits from this?” and “How does this help the user or business?”
  • Estimable: You must be able to roughly estimate the effort or complexity of the story.
  • Small: Small stories reduce risk, make progress visible, and give fast feedback.
  • Testable: Vague stories like “Improve performance” or “Make it look better” are hard to test.
Epic Title User Story Acceptance Criteria
Virtual Consultations US001: As a patient, I want to join a video call with my doctor via my computer, so I can receive care from home. Patient can initiate video call 5 minutes early; Provider is notified; Call is encrypted.

Task 3: Create a Product Backlog

Now we're ready to prioritize! Create a backlog using the user stories above - breaking them down further as needed.

  • Start with the MVP in mind. Focus on what's required for the 6-week MVP demo.
  • Your backlog should include more than just user stories. Make sure to include technical tasks, compliance tasks, accessibility reviews, etc. Just make sure it's trackable!
  • Use relative sizing (e.g., T-shirt sizes) for effort estimates. Estimation helps with sprint planning and capacity planning, not forecasting down to the hour.
